## 📋 Sobre o Projeto

### Descrição

Esta aplicação foi desenvolvida para gerenciar empreendimentos do estado de Santa Catarina, permitindo o cadastro, listagem, edição e exclusão de informações sobre negócios locais. A solução segue as melhores práticas de desenvolvimento, utilizando padrões de design modernos, código limpo e arquitetura em camadas.

### Funcionalidades Principais

- ✅ **CRUD Completo**: Criar, ler, atualizar e deletar empreendimentos
- ✅ **Validação de Dados**: Validação tanto no frontend quanto no backend
- ✅ **Interface Moderna**: UI responsiva e intuitiva com busca e filtros
- ✅ **API RESTful**: Endpoints bem estruturados seguindo padrões REST
- ✅ **Testes**: Cobertura de testes unitários e de integração
- ✅ **Tratamento de Erros**: Sistema robusto de tratamento de erros
- ✅ **Notificações**: Sistema de notificações toast para feedback ao usuário
- ✅ **Modal de Confirmação**: Modais customizados para ações importantes

### Modelo de Dados

A aplicação gerencia empreendimentos com os seguintes campos:

| Campo | Tipo | Obrigatório | Limite |
|-------|------|------------|--------|
| **Nome do Empreendimento** | String | Sim | Máx. 200 caracteres |
| **Nome do(a) Empreendedor(a)** | String | Sim | Máx. 200 caracteres |
| **Município de Santa Catarina** | String | Sim | Máx. 100 caracteres |
| **Segmento de Atuação** | Enum | Sim | Tecnologia, Comércio, Indústria, Serviços, Agronegócio |
| **E-mail ou Meio de Contato** | String | Sim | Máx. 200 caracteres |
| **Status** | Enum | Não | ativo (padrão) ou inativo |

```bash
echo "deb [ arch=amd64,arm64 signed-by=/usr/share/keyrings/mongodb-server-6.0.gpg ] https://repo.mongodb.org/apt/ubuntu jammy/mongodb-org/6.0 multiverse" | sudo tee /etc/apt/sources.list.d/mongodb-org-6.0.list

sudo apt-get install -y mongodb-org
sudo systemctl start mongod
```

**macOS**:

2. Crie uma conta gratuita

### 📦 Backup e Restore do MongoDB

O projeto inclui scripts para fazer backup e restaurar o banco de dados MongoDB facilmente.

#### Criar Backup

```bash
# Criar backup completo do banco de dados
npm run mongodb:backup
```

O backup será salvo em `data/backups/` com um timestamp no nome do arquivo (ex: `backup_20240115_143022.archive.gz`).

**O que o backup faz**:
- ✅ Cria um backup completo do banco de dados
- ✅ Comprime o backup com gzip para economizar espaço
- ✅ Salva na pasta `data/backups/`
- ✅ Mostra o tamanho do arquivo criado
- ✅ Lista os últimos 5 backups

#### Restaurar Backup

```bash
# Restaurar o backup mais recente
npm run mongodb:restore

# Restaurar um backup específico
npm run mongodb:restore data/backups/backup_20240115_143022.archive.gz
```

**⚠️ Atenção**: A restauração substitui todos os dados atuais do banco de dados!

#### Listar Backups Disponíveis

```bash
# Listar os últimos 10 backups
npm run mongodb:backup:list
```

#### Iniciar MongoDB com Restore Automático

```bash
# Inicia o MongoDB e restaura automaticamente o backup mais recente
npm run mongodb:start:with-restore
```

Este comando é útil para desenvolvimento, garantindo que você sempre comece com dados conhecidos.

#### Exemplos de Uso

**Backup antes de testes destrutivos**:
```bash
# 1. Criar backup
npm run mongodb:backup

# 2. Executar testes
npm test

# 3. Restaurar se necessário
npm run mongodb:restore
```

**Backup diário automático** (via crontab):
```bash
# Adicionar ao crontab para backup diário às 2h
0 2 * * * cd /caminho/do/projeto && npm run mongodb:backup
```

**Desenvolvimento com dados conhecidos**:
```bash
# Sempre iniciar com restore automático
npm run mongodb:start:with-restore
```

> 📖 Para documentação completa sobre backup e restore, consulte [MONGODB_BACKUP.md](./MONGODB_BACKUP.md).

## 🏗️ Arquitetura

### Visão Geral

A aplicação segue uma **arquitetura em camadas (Layered Architecture)** com separação clara de responsabilidades, facilitando manutenção, testes e escalabilidade.

```
┌─────────────────────────────────────────────────────────────┐
│                      FRONTEND (React)                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │   Pages      │  │  Components │  │   Services   │       │
│  │  (Views)     │  │ (Reusable)  │  │   (API)      │       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
└─────────────────────────────────────────────────────────────┘
                            │
                            │ HTTP/REST
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    BACKEND (Express)                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │  Routes      │  │ Controllers  │  │  Middleware  │       │
│  │  (Endpoints) │  │  (Request)   │  │ (Validation) │       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
│                            │                                 │
│                            ▼                                 │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │  Services    │  │ Repositories │  │    DTOs     │       │
│  │  (Business)  │  │   (Data)     │  │ (Transfer)   │       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    DATABASE (MongoDB)                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │   Models     │  │  Collections │  │   Indexes    │       │
│  │  (Schema)    │  │   (Data)     │  │ (Performance)│       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
└─────────────────────────────────────────────────────────────┘
```

### Camadas do Backend

#### 1. Camada de Rotas (Routes Layer)
- **Localização**: `backend/src/routes/`
- **Responsabilidade**: Define endpoints, mapeia URLs, aplica middlewares

#### 2. Camada de Controladores (Controllers Layer)
- **Localização**: `backend/src/controllers/`
- **Responsabilidade**: Recebe requisições HTTP, orquestra serviços, retorna respostas

#### 3. Camada de Serviços (Service Layer)
- **Localização**: `backend/src/services/`
- **Responsabilidade**: Lógica de negócio, validações, transformações

#### 4. Camada de Repositório (Repository Layer)
- **Localização**: `backend/src/repositories/`
- **Responsabilidade**: Acesso a dados, queries, abstração do banco

#### 5. Camada de Modelos (Models Layer)
- **Localização**: `backend/src/models/`
- **Responsabilidade**: Schema, validação, índices

### Camadas do Frontend

#### 1. Camada de Apresentação (Pages)
- **Localização**: `frontend/src/pages/`
- **Responsabilidade**: Componentes de página, orquestração

#### 2. Camada de Componentes (Components)
- **Localização**: `frontend/src/components/`
- **Responsabilidade**: Componentes reutilizáveis, UI

#### 3. Camada de Serviços (Services)
- **Localização**: `frontend/src/services/`
- **Responsabilidade**: Comunicação com API, transformação de dados

### Fluxo de Dados

```
Cliente → Frontend Service → Backend Route → Controller → 
Service → Repository → MongoDB → (resposta reversa)
```

### Por Que Esta Arquitetura é Melhor?

#### ✅ Separação de Responsabilidades
- Cada camada tem uma responsabilidade única
- Código mais fácil de entender e manter

#### ✅ Testabilidade
- Cada camada testável independentemente
- Mocks facilitados
- Cobertura de código alta (85%+)

#### ✅ Manutenibilidade
- Mudanças localizadas
- Código organizado
- Facilita code reviews

#### ✅ Escalabilidade
- Preparado para crescimento
- Fácil adicionar novas funcionalidades
- Suporta alta demanda

#### ✅ Reutilização
- Componentes e serviços reutilizáveis
- Menos código duplicado
- Consistência entre features

#### ✅ Desacoplamento
- Camadas independentes
- Fácil trocar tecnologias
- Flexibilidade

### Padrões de Design Aplicados

- **Repository Pattern**: Abstração de acesso a dados
- **Service Layer Pattern**: Centralização de lógica de negócio
- **DTO Pattern**: Tipagem forte entre camadas
- **Dependency Injection**: Baixo acoplamento
- **Middleware Pattern**: Cross-cutting concerns

## 📝 API - Documentação

### Base URL

```
http://localhost:3000/api
```

### Endpoints

#### `GET /empreendimentos`
Lista todos os empreendimentos.

**Resposta**:
```json
{
  "success": true,
  "data": [...],
  "count": 10
}
```

#### `GET /empreendimentos/:id`
Retorna um empreendimento específico.

**Resposta**:
```json
{
  "success": true,
  "data": {
    "_id": "...",
    "nomeEmpreendimento": "...",
    "nomeEmpreendedor": "...",
    "municipio": "...",
    "segmento": "...",
    "emailContato": "...",
    "status": "ativo"
  }
}
```

#### `POST /empreendimentos`
Cria um novo empreendimento.

**Body**:
```json
{
  "nomeEmpreendimento": "Tech Solutions",
  "nomeEmpreendedor": "João Silva",
  "municipio": "Florianópolis",
  "segmento": "Tecnologia",
  "emailContato": "joao@tech.com",
  "status": "ativo"
}
```

#### `PUT /empreendimentos/:id`
Atualiza um empreendimento existente (campos opcionais).

**Body**:
```json
{
  "nomeEmpreendimento": "Tech Solutions Updated",
  "status": "inativo"
}
```

#### `DELETE /empreendimentos/:id`
Deleta um empreendimento.

**Resposta**:
```json
{
  "success": true,
  "message": "Empreendimento deletado com sucesso"
}
```

#### ❌ Erro "Not supported URL scheme http+docker"

**Solução**: Use o método alternativo:
```bash
npm run mongodb:start:direct
```

#### ❌ MongoDB não conecta

**Soluções**:
1. Verifique se está rodando: `npm run mongodb:status`
2. Verifique a porta 27017
3. Verifique a URI no `.env`
4. Reinicie: `npm run mongodb:stop && npm run mongodb:start`

#### ❌ Container name already in use

**Solução**:
```bash
docker stop desafio-mongodb
docker rm desafio-mongodb
npm run mongodb:start
```

#### ❌ Porta 27017 já está em uso

**Soluções**:
1. Pare o MongoDB existente
2. Ou use outra porta (edite `docker-compose.dev.yml`)

> 💡 Para mais soluções detalhadas, consulte a seção completa de Troubleshooting no README original.
