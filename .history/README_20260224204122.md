# Desafio SC Tecia para Devs

Aplicação web full-stack com backend e frontend para gerenciamento de empreendimentos, implementando operações CRUD completas através de API REST.

## 📋 Descrição da Solução

Esta aplicação foi desenvolvida para gerenciar empreendimentos do estado de Santa Catarina, permitindo o cadastro, listagem, edição e exclusão de informações sobre negócios locais. A solução segue as melhores práticas de desenvolvimento, utilizando padrões de design modernos, código limpo e arquitetura em camadas.

### Funcionalidades Principais

- **CRUD Completo**: Criar, ler, atualizar e deletar empreendimentos
- **Validação de Dados**: Validação tanto no frontend quanto no backend
- **Interface Moderna**: UI responsiva e intuitiva
- **API RESTful**: Endpoints bem estruturados seguindo padrões REST
- **Testes**: Cobertura de testes unitários e de integração
- **Tratamento de Erros**: Sistema robusto de tratamento de erros

### Modelo de Dados

A aplicação gerencia empreendimentos com os seguintes campos:

- **Nome do Empreendimento**: Nome do negócio (obrigatório, máx. 200 caracteres)
- **Nome do(a) Empreendedor(a)**: Nome do responsável (obrigatório, máx. 200 caracteres)
- **Município de Santa Catarina**: Cidade onde o negócio está localizado (obrigatório, máx. 100 caracteres)
- **Segmento de Atuação**: Categoria do negócio (obrigatório)
  - Tecnologia
  - Comércio
  - Indústria
  - Serviços
  - Agronegócio
- **E-mail ou Meio de Contato**: Informação de contato (obrigatório, máx. 200 caracteres)
- **Status**: Estado do empreendimento (ativo ou inativo, padrão: ativo)

## 🏗️ Arquitetura da Aplicação

### Camadas do Backend

#### 1. **Camada de Rotas (Routes Layer)**
**Localização**: `backend/src/routes/`

**Responsabilidade**:
- Define os endpoints da API
- Mapeia URLs para controladores
- Aplica middlewares de validação

**Exemplo**:
```typescript
router.post('/', validate(createEmpreendimentoValidator), controller.create);
```

**Por que é melhor?**
- ✅ Separação clara de rotas por recurso
- ✅ Fácil adicionar novos endpoints
- ✅ Middlewares aplicados de forma declarativa
- ✅ Código organizado e legível

#### 2. **Camada de Controladores (Controllers Layer)**
**Localização**: `backend/src/controllers/`

**Responsabilidade**:
- Recebe requisições HTTP
- Extrai dados da requisição
- Chama serviços apropriados
- Retorna respostas HTTP formatadas

**Exemplo**:
```typescript
async create(req: Request, res: Response, next: NextFunction) {
  const data: CreateEmpreendimentoDto = req.body;
  const empreendimento = await service.create(data);
  res.status(201).json({ success: true, data: empreendimento });
}
```

**Por que é melhor?**
- ✅ Controladores são "finos" - apenas orquestram
- ✅ Lógica de negócio não fica misturada com HTTP
- ✅ Fácil testar isoladamente
- ✅ Reutilizável em diferentes contextos (REST, GraphQL, etc.)

#### 3. **Camada de Serviços (Service Layer)**
**Localização**: `backend/src/services/`

**Responsabilidade**:
- Contém toda a lógica de negócio
- Orquestra operações complexas
- Valida regras de negócio
- Transforma dados entre camadas

**Exemplo**:
```typescript
async create(data: CreateEmpreendimentoDto): Promise<EmpreendimentoResponseDto> {
  const empreendimento = await this.repository.create(data);
  return this.toResponseDto(empreendimento);
}
```

**Por que é melhor?**
- ✅ Lógica de negócio centralizada e reutilizável
- ✅ Fácil testar sem depender de HTTP ou banco
- ✅ Pode ser usado por diferentes interfaces (API, CLI, etc.)
- ✅ Mudanças na lógica não afetam rotas ou banco

#### 4. **Camada de Repositório (Repository Layer)**
**Localização**: `backend/src/repositories/`

**Responsabilidade**:
- Abstrai acesso ao banco de dados
- Encapsula queries complexas
- Fornece interface limpa para serviços

**Exemplo**:
```typescript
async findAll(): Promise<IEmpreendimento[]> {
  return await Empreendimento.find().sort({ createdAt: -1 });
}
```

**Por que é melhor?**
- ✅ Desacopla serviços do banco de dados
- ✅ Fácil trocar MongoDB por outro banco
- ✅ Queries complexas isoladas
- ✅ Testável com mocks

#### 5. **Camada de Modelos (Models Layer)**
**Localização**: `backend/src/models/`

**Responsabilidade**:
- Define estrutura dos dados
- Validação no nível do banco
- Índices para performance
- Relacionamentos entre entidades

**Por que é melhor?**
- ✅ Schema centralizado e validado
- ✅ TypeScript garante tipos corretos
- ✅ Validação automática do Mongoose
- ✅ Índices melhoram performance

### Camadas do Frontend

#### 1. **Camada de Apresentação (Pages)**
**Localização**: `frontend/src/pages/`

**Responsabilidade**:
- Componentes de página completos
- Orquestra componentes menores
- Gerencia estado da página

**Por que é melhor?**
- ✅ Páginas isoladas e independentes
- ✅ Fácil adicionar novas rotas
- ✅ Reutilização de componentes

#### 2. **Camada de Componentes (Components)**
**Localização**: `frontend/src/components/`

**Responsabilidade**:
- Componentes reutilizáveis
- Lógica de apresentação
- Interações do usuário

**Por que é melhor?**
- ✅ Componentes testáveis isoladamente
- ✅ Reutilização em múltiplas páginas
- ✅ Manutenção facilitada

#### 3. **Camada de Serviços (Services)**
**Localização**: `frontend/src/services/`

**Responsabilidade**:
- Comunicação com API
- Transformação de dados
- Tratamento de erros HTTP

**Por que é melhor?**
- ✅ Lógica de API centralizada
- ✅ Fácil mockar para testes
- ✅ Reutilizável em diferentes componentes

### Fluxo de Dados Completo

```
1. Cliente (Browser)
   ↓ HTTP Request
2. Frontend Service (api.ts)
   ↓ Axios
3. Backend Route (empreendimento.routes.ts)
   ↓ Middleware Validation
4. Backend Controller (Empreendimento.controller.ts)
   ↓ Extract Data
5. Backend Service (Empreendimento.service.ts)
   ↓ Business Logic
6. Backend Repository (Empreendimento.repository.ts)
   ↓ Mongoose Query
7. MongoDB Database
   ↓ Response
8. Repository → Service → Controller → Route → Frontend → Cliente
```

### Por Que Esta Arquitetura é Melhor?

#### 1. **Separação de Responsabilidades (SoC)**
Cada camada tem uma responsabilidade única e bem definida:
- **Rotas**: Apenas roteamento
- **Controladores**: Apenas orquestração HTTP
- **Serviços**: Apenas lógica de negócio
- **Repositórios**: Apenas acesso a dados

**Benefícios**:
- ✅ Código mais fácil de entender
- ✅ Mudanças isoladas não afetam outras camadas
- ✅ Fácil localizar bugs
- ✅ Onboarding de novos desenvolvedores mais rápido

#### 2. **Testabilidade**
Cada camada pode ser testada independentemente:

**Testes Unitários**:
- Services com mocks de repositories
- Repositories com mocks de models
- Controllers com mocks de services

**Testes de Integração**:
- Testam fluxo completo da API
- Verificam interação entre camadas

**Benefícios**:
- ✅ Testes rápidos e isolados
- ✅ Cobertura de código alta (85%+)
- ✅ Bugs detectados mais cedo
- ✅ Refatoração segura

#### 3. **Manutenibilidade**
Código organizado e fácil de modificar:

**Exemplo**: Para adicionar validação de email único:
- ✅ Adiciona validação no **Service** (lógica de negócio)
- ✅ Adiciona teste no **Service** (teste unitário)
- ✅ Nenhuma mudança em rotas, controllers ou repositories

**Benefícios**:
- ✅ Mudanças localizadas
- ✅ Código mais limpo
- ✅ Menos bugs ao modificar
- ✅ Facilita code reviews

#### 4. **Escalabilidade**
Arquitetura preparada para crescimento:

**Horizontal Scaling**:
- Services podem ser extraídos para microserviços
- Repositories podem usar diferentes bancos
- Frontend pode usar diferentes backends

**Vertical Scaling**:
- Cada camada pode ser otimizada independentemente
- Cache pode ser adicionado em qualquer camada
- Performance melhorada sem refatoração completa

**Benefícios**:
- ✅ Fácil adicionar novas funcionalidades
- ✅ Suporta crescimento da equipe
- ✅ Preparado para alta demanda

#### 5. **Reutilização**
Componentes e serviços reutilizáveis:

**Backend**:
- Services podem ser usados por diferentes controllers
- Repositories podem ser usados por diferentes services
- Middlewares aplicados em múltiplas rotas

**Frontend**:
- Componentes usados em múltiplas páginas
- Services usados por diferentes componentes
- Hooks customizados reutilizáveis

**Benefícios**:
- ✅ Menos código duplicado
- ✅ Consistência entre features
- ✅ Desenvolvimento mais rápido

#### 6. **Desacoplamento**
Camadas independentes e intercambiáveis:

**Exemplo**: Trocar MongoDB por PostgreSQL:
- ✅ Apenas Repository precisa mudar
- ✅ Service, Controller e Routes permanecem iguais
- ✅ Testes de Service continuam funcionando

**Benefícios**:
- ✅ Flexibilidade tecnológica
- ✅ Fácil experimentar novas tecnologias
- ✅ Menos impacto em mudanças

#### 7. **Padrões de Design Aplicados**

**Repository Pattern**:
- Abstrai acesso a dados
- Facilita testes com mocks
- Permite trocar banco de dados facilmente

**Service Layer Pattern**:
- Centraliza lógica de negócio
- Reutilizável em diferentes contextos
- Testável sem dependências externas

**DTO Pattern**:
- Tipagem forte entre camadas
- Validação de dados
- Documentação implícita da API

**Dependency Injection**:
- Dependências explícitas
- Fácil mockar em testes
- Baixo acoplamento

**Middleware Pattern**:
- Cross-cutting concerns (validação, erro, logging)
- Reutilizável em múltiplas rotas
- Pipeline de processamento

### Comparação com Arquiteturas Alternativas

#### ❌ Arquitetura Monolítica Simples (Não Recomendada)
```
Routes → Database
```
**Problemas**:
- Lógica de negócio misturada com rotas
- Difícil testar
- Código duplicado
- Difícil manter

#### ✅ Nossa Arquitetura em Camadas (Recomendada)
```
Routes → Controllers → Services → Repositories → Database
```
**Vantagens**:
- Separação clara de responsabilidades
- Fácil testar cada camada
- Código reutilizável
- Manutenção facilitada

#### ❌ Arquitetura sem Camadas (Não Recomendada)
Tudo em um único arquivo ou poucos arquivos grandes.

**Problemas**:
- Código difícil de navegar
- Mudanças afetam tudo
- Testes difíceis
- Onboarding lento

#### ✅ Nossa Arquitetura Modular (Recomendada)
Cada camada em seu próprio diretório com arquivos específicos.

**Vantagens**:
- Fácil encontrar código
- Mudanças isoladas
- Testes organizados
- Onboarding rápido

### Instalação do MongoDB

Você tem três opções para executar o MongoDB:

#### Opção 1: MongoDB via Docker (Recomendado - Mais Fácil)

Esta é a forma mais simples e recomendada. O MongoDB será executado em um container Docker.

1. **Certifique-se de ter Docker instalado**:
```bash
# Verificar se Docker está instalado
npm run docker:check

# Ou verificar manualmente
docker --version
```

Se não tiver Docker instalado, siga as instruções abaixo:

**Ubuntu/Debian**:
```bash
# Atualizar pacotes
sudo apt update

# Instalar Docker e Docker Compose
sudo apt install docker.io docker-compose

# Iniciar serviço Docker
sudo systemctl start docker
sudo systemctl enable docker

# Adicionar seu usuário ao grupo docker (para não precisar de sudo)
sudo usermod -aG docker $USER

# IMPORTANTE: Faça logout e login novamente, ou execute:
newgrp docker

# Verificar instalação
docker --version
docker-compose --version
```

**macOS**: 
Baixe e instale [Docker Desktop](https://www.docker.com/products/docker-desktop)

**Windows**: 
Baixe e instale [Docker Desktop](https://www.docker.com/products/docker-desktop)

**Nota**: Versões mais recentes do Docker usam `docker compose` (com espaço) ao invés de `docker-compose` (com hífen). Os scripts do projeto detectam automaticamente qual usar.

2. **Inicie o MongoDB usando Docker Compose**:
```bash
# No diretório raiz do projeto (os scripts detectam automaticamente docker compose ou docker-compose)
npm run mongodb:start

# Ou manualmente (se preferir):
# docker compose -f docker-compose.dev.yml up -d
# ou
# docker-compose -f docker-compose.dev.yml up -d
```

Isso irá:
- Baixar a imagem do MongoDB 6 (se não estiver instalada)
- Criar um container chamado `desafio-mongodb`
- Expor a porta 27017
- Criar volumes persistentes para os dados

3. **Verifique se o MongoDB está rodando**:
```bash
docker ps
```

Você deve ver o container `desafio-mongodb` na lista.

4. **Para parar o MongoDB**:
```bash
docker-compose -f docker-compose.dev.yml down
```

5. **Para parar e remover os volumes (apaga os dados)**:
```bash
docker-compose -f docker-compose.dev.yml down -v
```

#### Opção 2: MongoDB Instalado Localmente

1. **Instale o MongoDB**:

**Linux (Ubuntu/Debian)**:
```bash

# Adicione o repositório
echo "deb [ arch=amd64,arm64 signed-by=/usr/share/keyrings/mongodb-server-6.0.gpg ] https://repo.mongodb.org/apt/ubuntu jammy/mongodb-org/6.0 multiverse" | sudo tee /etc/apt/sources.list.d/mongodb-org-6.0.list

# Atualize e instale
sudo apt-get update
sudo apt-get install -y mongodb-org
```

**macOS**:
```bash
brew tap mongodb/brew
brew install mongodb-community@6.0
```

**Windows**:
Baixe o instalador MSI do [site oficial do MongoDB](https://www.mongodb.com/try/download/community)

2. **Inicie o MongoDB**:

**Linux (systemd)**:
```bash
sudo systemctl start mongod
sudo systemctl enable mongod  # Para iniciar automaticamente no boot
```

**macOS**:
```bash
brew services start mongodb-community@6.0
```

**Windows**:
O MongoDB será iniciado como serviço automaticamente após a instalação.

3. **Verifique se está rodando**:
```bash
mongosh --eval "db.version()"
```

### Troubleshooting - Problemas Comuns

#### Erro "docker-compose: not found" ou "docker compose: not found"

**Problema**: Docker ou Docker Compose não está instalado

**Soluções**:

1. **Verifique se Docker está instalado**:
```bash
npm run docker:check
```

2. **Instale Docker no Ubuntu/Debian**:
```bash
# Atualizar pacotes
sudo apt update

# Instalar Docker e Docker Compose
sudo apt install docker.io docker-compose

# Iniciar serviço Docker
sudo systemctl start docker
sudo systemctl enable docker

# Adicionar seu usuário ao grupo docker
sudo usermod -aG docker $USER

# IMPORTANTE: Faça logout e login novamente, ou execute:
newgrp docker

# Verificar instalação
docker --version
```

3. **Se usar versão nova do Docker** (que usa `docker compose` ao invés de `docker-compose`):
   - Os scripts do projeto (`npm run mongodb:start`) detectam automaticamente qual comando usar
   - Você não precisa se preocupar com isso, os scripts fazem isso automaticamente

#### Erro "Container name already in use" ou conflito de container

**Problema**: O container MongoDB já existe (criado anteriormente pelo método alternativo)

**Sintomas**:
- `Conflict. The container name "/desafio-mongodb" is already in use`
- `Error response from daemon: Conflict`

**Solução**:
Os scripts agora detectam automaticamente se o container já existe e está rodando. Se você ver essa mensagem, o script já resolveu automaticamente!

Se quiser remover o container existente e recriar:
```bash
# Parar e remover container existente
docker stop desafio-mongodb
docker rm desafio-mongodb

# Depois inicie novamente
npm run mongodb:start
```

#### Erro "Not supported URL scheme http+docker" ou incompatibilidade docker-compose

**Problema**: Incompatibilidade entre docker-compose 1.29.2 e Docker 28.x

**Sintomas**:
- `Not supported URL scheme http+docker`
- `Error while fetching server API version`
- `docker.errors.DockerException`

**Solução Automática**:
Os scripts detectam automaticamente esse problema e usam um método alternativo (docker run direto). Se ainda assim não funcionar, use:

```bash
# Método alternativo direto (sempre funciona)
npm run mongodb:start:direct
```

**Solução Permanente** (atualizar docker-compose - opcional):

**⚠️ IMPORTANTE**: O método alternativo (`npm run mongodb:start:direct`) já está funcionando perfeitamente! Você não precisa atualizar nada se estiver usando esse método.

Se ainda quiser corrigir o docker-compose para uso futuro:

```bash
# Opção 1: Instalar docker-compose-v2 (Ubuntu 22.04)
sudo apt update
sudo apt install docker-compose-v2

# Depois use: docker compose (com espaço) ao invés de docker-compose

# Opção 2: Baixar binário direto do GitHub (mais confiável)
sudo curl -L "https://github.com/docker/compose/releases/latest/download/docker-compose-$(uname -s)-$(uname -m)" -o /usr/local/bin/docker-compose
sudo chmod +x /usr/local/bin/docker-compose
docker-compose --version  # Verificar

# Opção 3: Instalar via pip (pode ter problemas)
pip3 install --upgrade docker-compose
```

**Recomendação**: Continue usando `npm run mongodb:start:direct` que é mais simples e confiável!

**Solução Temporária** (usar método alternativo):
```bash
# Use os scripts diretos que não dependem de docker-compose
npm run mongodb:start:direct   # Iniciar
npm run mongodb:stop:direct    # Parar
```

#### Erro "permission denied" ou "Error while fetching server API version"

**Problema**: Você não tem permissão para acessar o Docker daemon

**Sintomas**:
- `permission denied while trying to connect to the Docker daemon socket`
- `Error while fetching server API version`
- `Not supported URL scheme http+docker` (pode ser também problema de permissão)

**Solução Rápida**:
```bash
# Use o script auxiliar
npm run docker:fix-permissions

# Ou manualmente:
sudo usermod -aG docker $USER
newgrp docker
```

**Solução Detalhada**:

1. **Adicione seu usuário ao grupo docker**:
```bash
sudo usermod -aG docker $USER
```

2. **Ative as novas permissões** (escolha uma opção):
   - **Opção 1**: Execute `newgrp docker` no terminal atual
   - **Opção 2**: Faça logout e login novamente
   - **Opção 3**: Reinicie o computador

3. **Verifique se funcionou**:
```bash
docker ps
```

Se o comando acima funcionar sem `sudo`, as permissões estão corretas!

**Nota**: Se ainda não funcionar após `newgrp docker`, faça logout completo e login novamente.

#### MongoDB não conecta

**Problema**: Erro de conexão ao MongoDB

**Soluções**:

1. **Verifique se o MongoDB está rodando**:
```bash
# Com Docker
npm run mongodb:status

# Localmente
# Linux
sudo systemctl status mongod

# macOS
brew services list | grep mongodb
```

2. **Verifique a porta 27017**:
```bash
# Ver se a porta está em uso
lsof -i :27017  # macOS/Linux
netstat -ano | findstr :27017  # Windows
```

3. **Verifique a URI de conexão no `.env`**:
   - Deve ser: `mongodb://localhost:27017/desafioscteciaparadevs`
   - Se usar Docker, não precisa mudar nada
   - Se usar MongoDB Atlas, use a string de conexão fornecida

4. **Reinicie o MongoDB**:
```bash
# Docker
npm run mongodb:stop
npm run mongodb:start

# Local
# Linux
sudo systemctl restart mongod

# macOS
brew services restart mongodb-community@6.0
```

#### Docker não está instalado

**Solução**: Instale Docker seguindo as instruções:
- **Linux**: `sudo apt-get update && sudo apt-get install docker.io docker-compose`
- **macOS/Windows**: Baixe [Docker Desktop](https://www.docker.com/products/docker-desktop)

#### Porta 27017 já está em uso

**Problema**: Outro MongoDB ou aplicação está usando a porta 27017

**Soluções**:

1. **Pare o MongoDB existente**:
```bash
# Se for outro container Docker
docker stop $(docker ps -q --filter "publish=27017")

# Se for MongoDB local
# Linux
sudo systemctl stop mongod

# macOS
brew services stop mongodb-community@6.0
```

2. **Ou use outra porta** (edite `docker-compose.dev.yml`):
```yaml
ports:
  - "27018:27017"  # Use 27018 no host
```
E atualize o `.env`:
```env
MONGODB_URI=mongodb://localhost:27018/desafioscteciaparadevs
```

#### Erro "Cannot connect to MongoDB"

**Soluções**:

1. **Aguarde alguns segundos** após iniciar o MongoDB (ele precisa de tempo para inicializar)

2. **Verifique os logs**:
```bash
npm run mongodb:logs
```

3. **Teste a conexão manualmente**:
```bash
# Docker
docker exec -it desafio-mongodb mongosh --eval "db.version()"

# Local
mongosh --eval "db.version()"
```

#### Dados não persistem

**Problema**: Dados são perdidos ao reiniciar o MongoDB

**Solução**: Certifique-se de que os volumes do Docker estão configurados corretamente. O `docker-compose.dev.yml` já inclui volumes persistentes.

**Verificar volumes**:
```bash
docker volume ls | grep desafio
```

```bash

```

#### Frontend

1. Build do frontend:
```bash
cd frontend
npm run build
```

2. Os arquivos em `frontend/dist/` podem ser servidos por:
   - **Nginx**: Configure um servidor web para servir os arquivos estáticos
   - **Apache**: Similar ao Nginx
   - **Vercel/Netlify**: Deploy direto via CLI ou interface web
   - **AWS S3 + CloudFront**: Para hospedagem estática

**Exemplo de configuração Nginx**:
```nginx
server {
    listen 80;
    server_name seu-dominio.com;
    root /caminho/para/frontend/dist;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
}
```

### Opção 2: Deploy com Docker (Recomendado)

#### Criar Dockerfile para Backend

Crie `backend/Dockerfile`:
```dockerfile
FROM node:18-alpine

WORKDIR /app

COPY package*.json ./
RUN npm ci --only=production

COPY . .
RUN npm run build

EXPOSE 3000

CMD ["node", "dist/index.js"]
```

#### Criar Dockerfile para Frontend

Crie `frontend/Dockerfile`:
```dockerfile
FROM node:18-alpine as build

WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]
```

#### Criar docker-compose.yml

No diretório raiz:
```yaml
version: '3.8'

services:
  mongodb:
    image: mongo:6
    container_name: mongodb
    ports:
      - "27017:27017"
    volumes:
      - mongodb_data:/data/db
    environment:
      MONGO_INITDB_DATABASE: desafioscteciaparadevs

  backend:
    build: ./backend
    container_name: backend
    ports:
      - "3000:3000"
    environment:
      NODE_ENV: production
      PORT: 3000
      MONGODB_URI: mongodb://mongodb:27017/desafioscteciaparadevs
      CORS_ORIGIN: http://localhost
    depends_on:
      - mongodb

  frontend:
    build: ./frontend
    container_name: frontend
    ports:
      - "80:80"
    depends_on:
      - backend

volumes:
  mongodb_data:
```

#### Deploy com Docker

```bash
docker-compose up -d
```
